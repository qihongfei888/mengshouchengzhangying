# Bmob SDK 版本与多端同步说明

## 一、当前项目中的 SDK 文件

| 文件 | 位置（相对 class 目录） | 说明 |
|------|------------------------|------|
| Bmob-1.6.1.min.js | ../ | 旧版 |
| Bmob-1.7.1.min.js | ../ | 旧版，API 与 2.x 不一致 |
| Bmob-2.5.0.min.js | ../ | 2.x |
| Bmob-2.5.2.min.js | ../ | 2.x |
| Bmob-2.5.20.min.js | ../ | 2.x |
| **Bmob-2.5.30.min.js** | ../ | **推荐使用** |
| Bmob-2.6.3.min.js | ../ | 2.x 可用 |
| Bmob-2.7.0.min.js | 若存在 | 易出现 415 参数类型错误，不推荐 |

## 二、导致“不能自动同步、多端数据不同步”的原因

1. **index.html 之前引用的是 Bmob-1.7.1.min.js**
   - 1.7.1 与 2.x 的 Query/equalTo/find/get/set 等 API 不一致。
   - app.js 里的同步逻辑是按 **2.x** 写的，用 1.7.1 时可能静默失败或报错，导致拉取/上传失败，多端不同步。

2. **部分环境使用 Bmob-2.7.0.min.js**
   - 2.7.0 在部分请求下会返回 **415 incorrect parameter type**（例如带 order/limit 或某些 equalTo 的请求）。
   - 同步请求失败后，云端拉取不到最新数据，多端就会不一致。

3. **SDK 脚本路径错误**
   - index.html 在 **class** 目录下，若写 `src="Bmob-1.7.1.min.js"` 且 Bmob 实际在**上一级目录**，则脚本 404，Bmob 未加载，所有同步都会失败。
   - 需要保证页面能加载到同目录的 `Bmob-2.5.30.min.js`，或使用 `../Bmob-2.5.30.min.js`（当文件在上一级时）。

## 三、已做的修改

- **统一 SDK 版本**：index.html 改为加载 **Bmob-2.5.30.min.js**（与 app.js 同步逻辑兼容，且避免 2.7.0 的 415）。
- **加载方式**：先尝试同目录 `Bmob-2.5.30.min.js`，若未加载成功则再尝试 `../Bmob-2.5.30.min.js`，兼顾“class 为站点根目录”和“Bmob 在上一级”两种部署方式。
- **部署时**：请将 **Bmob-2.5.30.min.js** 放在与 **index.html 同一目录**（例如一起放在 class 或站点根目录），否则请修改 index.html 中 script 的 src 为 `../Bmob-2.5.30.min.js`。

## 四、使用与排查建议

- 部署到 GitHub Pages 或其他静态托管时，确保 **Bmob-2.5.30.min.js** 和 **index.html** 在同一目录，或按实际目录改好 script src。
- 打开页面后看控制台：出现 “✅ Bmob SDK 加载成功，版本: 2.5.30” 表示 SDK 正常；若出现 “❌ Bmob SDK 未加载”，说明路径不对或文件缺失，需按上面方式调整路径或复制文件。
- 同步仍异常时，看控制台是否有 “Bmob同步失败” 或 415 等错误；若出现 415，当前 app.js 已对 415 做了降级拉取，若仍不行可考虑统一改用 2.5.30 并确认未再引用 2.7.0。
